Building a custom embedded Linux distribution with Yocto

  • Published 2022-06-19

Introduction

Yocto is an incredibly powerful tool for creating embedded Linux distributions. In this post, I'll discuss why I love it, and guide you through building a minimal image for the open hardware BeagleBoard X15

There are a few things I love about Yocto.

The first is that it provides a really great cross-compilation platform. It's 2022, don't build your software on-target. Use a powerful build server to produce an image with Yocto.

Cross-compliation can be a huge pain in the ass. Yocto makes the pain stop. Never worry about host pollution again.

Yocto also helps you manage the configuration of your Linux distribution. "Recipes" for software in the distribution are specified with git commit hashes. Tracking the recipes themselves on git allows you to know exactly what software is in a release, and rebuild the release whenever necessary.

Nothing in life is free, and Yocto has its costs too. Setting up a Yocto project can be complicated. At work, there are currently a handful of people who have fallen into "maintainer" roles. However, I think with a little bit of hands-on experience, most developers will be able to confidently create new recipes from existing ones.

Getting started

Docker setup

In the following git repository you'll find a Dockerfile describing the dependencies for the Yocto project.

The Dockerfile assumes your uid/gid is 1000:1000.

The following two steps are optional. If you want, you can skip them, and use the Dockerfile as a guide to set up your local environment. If you run into issues, I'd love to hear about it so I can fix these instructions. My email is andrew.hepp at this domain.

To build the Docker image, run: 

    $ docker build . -t $X15_IMAGE
where $X15_IMAGE is your preferred name for the image.

Next, instantiate a container based on the image. Bind mount the host system's working directory to the container path /ws. Optionally, mount a Docker volume at /ws/x15 to hold the source and build directories. I did this to work around some issues with Docker for MacOS (specifically, the case sensitive native filesystem). 

    $ docker run -it -v $(pwd):/ws -v $X15_VOLUME:/ws/x15 $X15_IMAGE
where $X15_VOLUME is your preferred name for the volume.

Fetching sources and configuring Yocto

The following steps can be done automatically with ./scripts/fetch.sh

Once the build environment is set up, you need to fetch poky, the Yocto reference distribution; and meta-ti, the "layer" to support our TI based board. meta-ti depends in turn on meta-arm, so we'll fetch that too. We'll store these layers in /ws/x15/sources. Note that this is not the source for our custom software, it's the source for the recipes on how to build that software. 

    $ mkdir -p /ws/x15/sources
    $ git clone -b kirkstone git://git.yoctoproject.org/poky     /ws/x15/sources/poky
    $ git clone -b kirkstone git://git.yoctoproject.org/meta-ti  /ws/x15/sources/meta-ti
    $ git clone -b kirkstone git://git.yoctoproject.org/meta-arm /ws/x15/sources/meta-arm

Our build will be done in /ws/x15/build. We need to create a couple configuration files in /ws/x15/build/conf. First we need bblayers.conf, describing the layers Yocto will use. 

/ws/x15/build/conf/bblayers.conf
    POKY_BBLAYERS_CONF_VERSION = "2"

    BBPATH = "${TOPDIR}"
    BBFILES ?= ""

    BBLAYERS ?= " \
      /ws/x15/sources/poky/meta \
      /ws/x15/sources/poky/meta-poky \
      /ws/x15/sources/meta-arm/meta-arm \
      /ws/x15/sources/meta-arm/meta-arm-toolchain \
      /ws/x15/sources/meta-ti/meta-ti-bsp \
      "
will do

Next we need local.conf, describing configuration options for the entire build. We'll only add a version number and a string telling Yocto what our machine is. 

/ws/x15/build/conf/local.conf
    CONF_VERSION = "2"                                                                
    MACHINE="beagle-x15"

Building an image

The hard part is done. As with fetching, we could simply run ./scripts/build.sh. If you want to see how the sausage is made, source the openembedded setup script, telling it our build directory: The final product can be found here. This file is an uncompressed tar archive containing a wic.xz and bmap. Note that this image will not boot without the uEnv.txt mentioned at the end of this article. 

    $ . /ws/x15/sources/poky/meta/oe-init-build-env /ws/x15/build

You should now be able to run 

    $ bitbake core-image-minimal

The build will take an hour or more even on a reasonably powerful dev machine. On an AWS c6g4xlarge, it takes about 45 minutes. When it completes, you'll have a file /ws/x15/build/tmp/deploy/images/beagle-x15/core-image-minimal-beagle-x15.wic.xz, and an associated bmap file. 

    bmaptool copy core-image-minimal-beagle-x15.wic.xz /dev/mmcblk0
will use the XML description in the bmap file to format the sd card at /dev/mmcblk0. As always take care that this device is actually the one you intend to format.

Unfortunately the image as created did not boot for me. u-boot seemed very confused about the MMC device and partition it was supposed to boot from, as well as the root device. I had to mount /dev/mmcblk0p1 and add the following uEnv.txt: 

/mnt/uEnv.txt
    mmcdev=0
    bootpart=0:1
    bootdir=
    args_mmc=run finduuid;setenv bootargs console=${console} ${optargs} root=/dev/mmcblk0p2 rw rootfstype=${mmcrootfstype}

Figuring out the best way to fix this will be a future project. It's not immediately clear to me whether this should be patched in meta-ti, or dealt with in a custom layer. I haven't messed with the bootloader on this board, so I expected the image to boot out of the box. Anyways, it gave me the chance to learn a bit about u-boot, so I won't complain too much.
